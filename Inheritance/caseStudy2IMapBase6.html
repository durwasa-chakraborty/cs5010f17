The containsKey and get Methods

The abstract help methods are used by three public methods that are listed in the Map<K,V> interface, and are available for use in subclasses of the base class. 

          // public methods listed by Map<K,V> interface
      
          // Returns true iff this map contains a mapping for the given key.
      
          @Override
          @SuppressWarnings("unchecked")
          public boolean containsKey (Object x) {
              if (x == null) {
                  String nullMsg = "null values aren't allowed by IMap<K,V>";
                  throw new NullPointerException (nullMsg);
              }
              else {
                  K key = (K) x;
                  return hasKey (key);
              }
          }
      
          // Returns the value associated with the given key,
          // or null if the key is not contained within the map.
      
          @Override
          @SuppressWarnings("unchecked")
          public V get (Object x) {
              if (containsKey (x)) {
                  K key = (K) x;
                  return getVal (key);
              }
              else return null;
          }

These methods override methods defined in the AbstractMap<K,V> class.

For historical reasons, these methods are specified to take an arbitrary Object as their argument instead of a key of type K. That forces them to perform an unchecked cast, which causes the Java compiler to issue a warning, which should be suppressed before an excessive number of routine warning messages makes it too easy to overlook an important warning message.

The containsKey and get methods also take care of some annoying details we don't want to re-implement in every subclass:

  • The containsKey method throws an exception if the key is null, which enforces an invariant present in the IMap<K,V> interface.
  • The get method is specified to return null if its argument is not a key of this map. Implementing that behavior in the base class is convenient because it doesn't have to be implemented in subclasses, but it results in two separate lookups per get operation: one to see whether the key is present, and another to fetch its associated value. If that factor of two becomes important, subclasses can override the get method.
