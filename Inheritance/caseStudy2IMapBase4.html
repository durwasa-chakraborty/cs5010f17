Design Strategy for the Abstract Base Class

The design strategy for our abstract base class is the same as the strategy used for the IMapList<K,V> class in a previous lesson: 

      // Strategy, quoted from the documentation for AbstractMap:
      //
      // To implement an unmodifiable map, the programmer needs only to
      // extend [the AbstractMap] class and provide an implementation
      // for the entrySet method, which returns a set-view of the map's
      // mappings.  Typically, the returned set will, in turn, be
      // implemented atop AbstractSet.  This set should not support the
      // add or remove methods, and its iterator should not support the
      // remove method.
      
      abstract class IMapBase<K,V> extends AbstractMap<K,V> implements IMap<K,V> {
      
          // Java constructor
      
          protected IMapBase () { }
      
          ...
      }

Why the Constructor Should Be protected

The Java constructor shouldn't be public, because it should only be used by static methods in the IMaps class and by subclasses of the base class.

On the other hand, we want to make the Java constructor accessible within all subclasses, even if those subclasses aren't defined in the same package.

If we were to omit its access modifier, the constructor wouldn't be accessible in subclasses outside of the package. That's too much protection.

To make the constructor less protected, we declare it protected.
